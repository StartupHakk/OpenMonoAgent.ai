openmono.ai · Setup-Guide

Vulkan-Lane starten

llama-server läuft auf deiner Radeon 780M iGPU — mit System-RAM als VRAM. Zwei Befehle, fertig.

CPU Ryzen 9 PRO 8945HS iGPU Radeon 780M GTT-Fenster 96 GB Kernel-Parameter ✓ aktiv
1

Einmalig: Setup mit Vulkan-Option

$ cd /workspace/openmono.ai
$ ./openmono setup --vulkan

Das schreibt die Vulkan-Konfiguration nach docker/docker-compose.override.yml (Image llama.cpp:server-vulkan, iGPU-Zugriff über /dev/dri, 32 GB shm).

Kein Reboot nötig. Deine Kernel-Parameter (amdgpu.gttsize=98304) sind bereits aktiv — das Setup erkennt das und überspringt die Kernel-Optimierung. Dein Modell liegt schon in models/, es wird nichts neu heruntergeladen.

Alternativ ohne Flag: openmono setup erkennt deinen 8945HS jetzt automatisch und fragt — dann 2) iGPU (Vulkan) wählen.

2

Entwicklungsumgebung öffnen

$ cd dein-projekt/
$ openmono agent

Mehr ist es nicht: openmono agent startet llama-server automatisch über die Vulkan-Lane, wartet bis er bereit ist, und öffnet dann den Agent im aktuellen Ordner.

Der erste Start dauert 1–2 Minuten — das 17,7-GB-Modell wird ins GTT-Fenster geladen. Danach bleibt der Server im Hintergrund laufen, weitere Starts sind sofort da.

Modelle wechseln

36 Modelle liegen bereit (aus dem Model-Shelf verlinkt). openmono model listet sie und schaltet auf Knopfdruck um — .env und Server werden automatisch angepasst.

    GLM-4.6V-Flash-UD-Q4_K_XL.gguf                 5,8G

    QwQ-32B-UD-Q3_K_XL.gguf                         16G
  ▶ Qwen3.6-35B-A3B-UD-Q4_K_XL.gguf                 21G
    …  (openmono model  zeigt alle 36)
$ openmono model use dolphin
[OK] Model set: Dolphin3.0-Mistral-24B-Q4_K_M.gguf
     alias: …  ·  vision: off  ·  ctx: 204800
[OK] llama-server is healthy — model 'Dolphin…' is live

Der Name darf ein Teilstück sein (Groß/klein egal) — glm-4.6v, phi-4, qwen3.6-27b. Bei mehrdeutigem Namen zeigt der Befehl die Treffer an. Vision-Modelle bekommen ihren Projektor automatisch dazu, Text-Modelle laufen ohne.

Große Modelle knapp im GTT-Fenster? Kontext beim Umschalten kleiner setzen: openmono model use qwen3-235b --ctx 65536. Lädt ein Modell nicht (OOM oder Absturz), ist das der erste Hebel.
Nicht jedes GGUF passt: Diffusion-Modelle (DiffuCoder, Dream-Coder, diffusiongemma – Architektur dream) brauchen einen speziellen Runner und stürzen auf dem normalen Server ab. Modelle über ~90 GB sprengen das 96-GB-GTT-Fenster.

Doom-Loop-Detection ausschalten

Der Agent bricht ab, wenn er dieselben Tool-Aufrufe 3× hintereinander macht — Schutz vor Endlosschleifen. Bei repetitiven Aufgaben wie Übersetzungen ist genau das aber gewollt. Dafür gibt es /doomloop.

$ openmono agent
› /doomloop off
Doom loop detection OFF — repetitive tool calls will NOT be aborted.
Useful for repetitive work like translations. Re-enable with /doomloop on.

Der Befehl wird im Agent eingetippt (nicht im Terminal davor). Danach die Übersetzung ganz normal anstoßen — sie läuft jetzt ohne Abbruch durch.

BefehlWirkung
/doomloop offDetection aus (auch: aus, false, 0)
/doomloop onDetection wieder an (auch: an, true, 1)
/doomloopohne Argument: umschalten (Toggle)
Gilt pro Session. Nach einem Neustart von openmono agent ist die Detection wieder an — bewusst so, damit der Schutz nicht versehentlich dauerhaft aus bleibt. Für jede neue Übersetzungs-Session also einmal /doomloop off eintippen.

Gut zu wissen

Wenn du …dann …
zurück zu CPU oder CUDA willst openmono setup --cpu bzw. --gpu — die Lanes überschreiben sich nicht, es zählt immer das zuletzt geschriebene Override.
die Kontextgröße ändern willst docker/.env → CTX_SIZE=… setzen, dann openmono restart. Aktuell: 204800 (mit Vision), bis 262144 möglich.
ein weiteres GGUF hinzufügen willst Datei nach models/ hardlinken (ln quelle.gguf models/, kein Symlink — der bricht im Container), dann taucht es in openmono model auf.
„Doom loop detected“ eine Aufgabe abbricht Im Agent /doomloop off eintippen (siehe Abschnitt oben) — bei repetitiven Aufgaben wie Übersetzungen ist die Wiederholung gewollt.
„out of memory“ im Log siehst Kleinere Quantisierung wählen (Q4_K_M → Q3_K_M / IQ3_XXS) oder CTX_SIZE reduzieren.
auf einem frischen System installierst Das Setup richtet die Kernel-Parameter selbst ein und fragt nach einem Reboot — danach einfach openmono setup erneut ausführen.
Updates vom offiziellen Repo holen willst Updates sind absichtlich blockiert — deine Vulkan-Anpassungen liegen auf dem lokalen Branch vulkan-lane, und re-install.sh bricht bei lokalen Änderungen ab. Bewusst updaten: git fetch origin && git rebase origin/main — deine Commits bleiben dabei erhalten.
